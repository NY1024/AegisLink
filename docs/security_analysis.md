# AegisLink 领域认知与技术白皮书

## 1. 为何传统IAM无法直接套用于AI Agent场景

### 1.1 身份模型的根本差异

| 维度 | 传统IAM | AI Agent |
|-----|-------|----------|
| 身份来源 | 人类用户，通过凭证(用户名/密码、证书) | Agent本身，需要可验证的数字身份 |
| 行为确定性 | 高(人类操作可预期) | 低(LLM输出具有随机性) |
| 授权范围 | 静态、预定义 | 动态、上下文相关 |
| 信任建立 | 基于网络边界+凭证 | 需要可验证的信任链(Chain of Trust) |
| 调用模式 | 点对点 | 多跳、多Agent协作链 |

### 1.2 传统IAM的局限性

1. **调用链路的复杂性**: 传统IAM处理User→App的二维关系，而Agent场景是User→Agent A→Agent B→Service C的四维关系。

2. **身份的可变性**: 人类身份相对稳定，而Agent身份可能因模型更新、Prompt变更而表现出不同行为。

3. **权限的动态性**: 传统IAM权限在会话期间相对固定，而Agent需要根据任务上下文动态计算权限。

4. **信任传递问题**: 在多Agent协作中，信任无法通过简单的凭证传递，需要可验证的信任链。

## 2. Agent身份模型与人类用户身份的区别

### 2.1 身份持久性
- **人类用户**: 身份与生物特征绑定，长期稳定
- **Agent**: 身份与配置(Prompt、模型、工具集)绑定，可能随版本迭代变化

### 2.2 行为可预测性
- **人类用户**: 行为具有法律约束，可追溯
- **Agent**: 行为由概率模型决定，存在不确定性

### 2.3 身份验证方式
- **人类用户**: 知识凭证(密码)或生物特征
- **Agent**: 数字签名(JWT)、能力声明(Capability Statement)

## 3. Prompt Injection等AI特有攻击对权限系统的威胁

### 3.1 Prompt Injection攻击原理
攻击者通过在输入中注入恶意指令，诱导LLM忽略原始指令或执行未授权操作。

### 3.2 对权限系统的威胁

1. **指令覆盖**: 恶意指令可能覆盖系统的权限检查指令
2. **上下文混淆**: 攻击者可利用Agent的信任关系获取未授权信息
3. **能力滥用**: 即使有细粒度权限控制，Prompt Injection可能导致Agent"自愿"执行越权操作

### 3.3 AegisLink的防御措施

1. **Token强制校验**: 所有Agent间调用必须携带有效Token，系统强制校验
2. **能力声明验证**: 调用方必须声明意图(target_agent_id, action)，系统验证能力匹配
3. **短时Token**: 使用临时调用Token(10分钟有效期)限制暴露窗口
4. **实时撤销**: 支持Token撤销机制，可立即失效被滥用的Token

## 4. 主流Agent框架在IAM层面的不足分析

### 4.1 主流框架概览

| 框架 | 身份机制 | 权限控制 | 审计能力 |
|-----|---------|---------|---------|
| LangChain | 无原生身份 | 无细粒度权限 | 无原生审计 |
| AutoGPT | 无身份验证 | 无权限概念 | 无审计 |
| CrewAI | 基础角色定义 | 无细粒度控制 | 无 |
| Microsoft AutoGen | 无 | 无 | 无 |

### 4.2 共同缺陷

1. **缺乏统一身份标准**: 各框架使用自己的Agent标识方式，无法互操作
2. **权限控制粗粒度**: 多为基于角色的简单控制，缺乏能力声明
3. **无信任链机制**: Agent间调用无验证机制
4. **审计日志缺失**: 无法追溯决策来源
5. **无法对抗Prompt Injection**: 无防护机制

### 4.3 改进思路

1. **标准化身份层**: 定义Agent身份规范，类似X.509证书的Agent Identity Certificate
2. **能力声明标准**: 制定Capability Statement规范，支持权限的跨平台验证
3. **信任链协议**: 实现可验证的Chain of Trust，类似TLS握手过程
4. **实时审计**: 建立集中式审计日志，支持溯源和告警
5. **动态权限计算**: 根据上下文(用户意图、任务需求)动态计算最小权限

## 5. AegisLink设计决策说明

### 5.1 为何选择JWT而非证书
JWT轻量且自包含，适合Agent间的快速验证场景。证书更适合长期身份验证。

### 5.2 为何采用能力声明而非RBAC
能力声明(Capability)更细粒度，可精确描述"能做什么"，适合动态授权场景。

### 5.3 信任链的实现
通过trust_chain字段记录每次调用的证明，支持完整的溯源。

### 5.4 动态授权的价值
临时Token有10分钟有效期，限制Token被盗用的影响窗口。
