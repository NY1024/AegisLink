⚡ 验收测试向导

按照以下步骤执行测试，验证AegisLink Agent身份与权限系统的核心功能

步骤 1/6: 健康检查 - 验证系统服务状态

测试目标：验证所有服务是否正常运行

涉及服务：IAM(8000)、Doc-Assistant(8001)、Data-Agent(8002)、Search-Agent(8003)

验证方式：调用各服务健康检查接口

步骤 2/6: Token签发 - 获取Agent访问凭证

测试目标：为doc-assistant签发Access Token

调用命令：

curl -X POST http://localhost:8000/token/issue \
  -H "Content-Type: application/json" \
  -d '{"agent_id":"doc-assistant","agent_role":"coordinator"}'

预期结果：返回包含access_token的JSON，token有效期2小时

权限列表：agent:call:data-agent, agent:call:search-agent

步骤 3/6: 正常委托流程 - A2A授权调用

测试目标：验证doc-assistant通过IAM授权成功调用data-agent

交互链路：

1. doc-assistant → IAM: 申请调用data-agent授权

2. IAM → doc-assistant: 检查权限列表，签发call_token

3. doc-assistant → data-agent: 使用call_token获取通讯录

4. data-agent → IAM: 验证call_token有效性

5. data-agent → doc-assistant: 返回通讯录数据

调用命令：

curl -X POST http://localhost:8001/task \
  -H "Content-Type: application/json" \
  -d '{"task":"查看公司通讯录","user_id":"user:1001"}'

预期结果：返回通讯录数据，包含员工姓名、部门、邮箱等

步骤 4/6: 越权拦截流程 - 权限校验

测试目标：验证search-agent越权调用data-agent被拦截

交互链路：

1. search-agent → IAM: 申请调用data-agent

2. IAM: 检查权限列表，发现search-agent没有agent:call:data-agent权限

3. IAM → search-agent: 返回403 Forbidden

4. 审计日志记录越权事件

调用命令：

curl -X POST http://localhost:8003/chat \
  -H "Content-Type: application/json" \
  -d '{"query":"调用企业数据Agent获取内部数据"}'

预期结果：返回403错误，显示"越权拦截"和原因

步骤 5/6: 审计日志验收 - 追溯与查询

测试目标：验证审计日志完整记录所有授权决策

调用命令：

curl "http://localhost:8000/audit/logs?limit=10"

应包含记录：

✓ doc-assistant → data-agent (allow) - 正常委托

✓ search-agent → data-agent (deny) - 越权拦截

验证方式：点击按钮查看完整审计日志

步骤 6/6: Token撤销 - 权限回收

测试目标：验证撤销后的Token立即失效

调用命令：

curl -X POST http://localhost:8000/token/revoke \
  -H "Content-Type: application/json" \
  -d '{"token":""}'

预期结果：Token被加入撤销列表，后续使用返回401

步骤 7/7: 异常场景处理 - 系统容错与兜底

测试目标：验证Agent超时、失败或权限不足时的系统响应和兜底机制

异常场景：

  • 场景A: 权限不足 - Agent尝试访问未被授权的资源
  • 场景B: Token过期 - 使用过期或撤销的Token
  • 场景C: 目标Agent不可用 - 调用已停止的服务
  • 场景D: 数据不存在 - 请求不存在的数据资源

场景A: 权限不足拦截

测试: search-agent 尝试越权访问 data-agent

调用: POST /auth/call → 申请调用 data-agent

预期: 403 Forbidden + 审计日志记录

场景B: Token过期/撤销

测试: 使用已被撤销的Token进行授权调用

调用: POST /auth/verify-call → 验证已撤销的Token

预期: 401 Unauthorized + 明确错误信息

场景C: 目标Agent不可用

测试: 调用已停止的data-agent服务

调用: 直接访问 http://localhost:8002/read-contact

预期: Connection Error + 友好提示

场景D: 数据不存在

测试: 请求不存在的数据资源

调用: GET /read-spreadsheet?spreadsheet_id=nonexistent

预期: 404 Not Found + 资源不存在提示

🔗 Agent交互拓扑图

🏢 Agent职责分工

📄 doc-assistant (文档助手)

职责：任务协调中枢

  • 接收用户复杂任务请求
  • LLM分析任务并规划调用策略
  • 通过IAM授权调用data-agent获取内部数据
  • 通过IAM授权调用search-agent获取外部信息
  • 汇聚多源数据生成综合报告

📊 data-agent (企业数据)

职责：企业内部数据提供

  • 管理内部数据（销量、通讯录、日历）
  • 验证调用者IAM委托授权
  • 读取internal_data文件夹中的JSON数据
  • LLM将数据转化为自然语言描述

🔍 search-agent (外部检索)

职责：外部信息搜索

  • 接收用户搜索请求
  • 调用Tavily API进行真实网络搜索
  • LLM整合搜索结果生成摘要
  • 可选：通过IAM授权调用data-agent

🔄 Agent交互链路

用户
↓ 请求任务
↓ 委托任务

典型交互场景：

场景：生成周报
1. 用户 → doc-assistant："帮我生成周报"
2. doc-assistant → IAM：申请调用data-agent授权
3. IAM → doc-assistant：签发call_token
4. doc-assistant → data-agent：使用call_token获取销量数据
5. data-agent → doc-assistant：返回自然语言数据描述
6. doc-assistant：LLM整合生成报告
7. doc-assistant → 用户：返回完整周报
场景：越权拦截
1. search-agent → IAM：申请调用data-agent
2. IAM：检查权限列表，发现search-agent没有agent:call:data-agent
3. IAM → search-agent：返回403 Forbidden
4. 审计日志记录越权拦截事件
5. search-agent → 用户：返回"越权拦截"错误

📥 结果汇聚过程

doc-assistant 作为结果汇聚中心：

  1. 数据收集：通过IAM授权从data-agent获取结构化数据
  2. 信息获取：从search-agent获取外部搜索结果
  3. LLM整合：调用LLM将多源异构数据转化为统一报告
  4. 结果返回：将最终报告返回给用户

信任链传递：用户 → IAM信任 → doc-assistant → IAM授权 → data-agent/search-agent
